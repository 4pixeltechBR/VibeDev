# Release v3.2.0 — Layman Mode Quality of Life

🌍 **Read in / Leia em:** [English](#-english) | [Português](#-português)

---

## 🇺🇸 English

**Release date:** 2026-07-09
**Scope:** VibeDev core (v1.3.0)

In v3.1.0 we solved three "I don't understand" problems for laymen (jargon, cost, kill). This release v3.2.0 solves three "I'm exhausted / I lost control / I don't know how much longer" problems.

These features are passive — they surface only when relevant. They don't change any existing command flow; they add behavior to existing commands.

### What's in this release

#### 1. Calibrated Construction-Time Estimates

New file: `vibedev/references/tempo-construcao.md`

Loaded during `/vd-start` in layman mode **before** formal diagnosis. The 9 most common project categories get a realistic time range:

| Category | Layman (4h+/day) | Layman (spare time) | Dev |
|---|---|---|---|
| Landing page (1 static page) | 1-3 days | 1-2 weeks | 2-8h |
| Institutional site (3-10 pages) | 2-4 weeks | 1-2 months | 1-3 weeks |
| Internal app / private dashboard | 1-2 months | 2-6 months | 2-6 weeks |
| Messenger bot (Telegram/WhatsApp + LLM) | 3-8 weeks | 2-4 months | 1-2 weeks |
| Public B2C SaaS (freemium, login, payment) | 3-12 months | 6-24 months | 2-6 months |
| Mobile app (iOS + Android store) | rarely finishes | rarely finishes | 3-9 months |
| Marketplace (Uber/iFood style) | don't try alone | don't try alone | 6-18 months |
| AI-generative app (chatbot, image gen) | 1-4 months (MVP) | 2-8 months | 2-6 weeks |

**Honest framing:** "people with ideas similar to yours generally take X to Y weeks working Z hours per day. Not a firm rule — it's so you don't think you're falling behind when you're not."

**Leak mitigation:** if the layman has Instagram-reel timeline expectations and clearly not the time, the framework surfaces `/vd-kill` as a legitimate option **before** they start, not 90 days into a stalled project.

#### 2. Anti-Feature-Creep Guard

New file: `vibedev/references/anti-creep.md`

Three-layer containment that keeps the scope honest:

- **Layer 1 — Pre-build scope check:** before writing code for a sub-task, the IA compares intent with `anti-scope`, done-criterion of the active sub-task, and the remaining sub-task list. Mismatch → backlog, not code.
- **Layer 2 — Real-time detection:** new ideas during `/vd-build` (whether from IA or user) are recognized as good, registered in the backlog automatically, and execution continues on the planned sub-task. End of sub-task mentions what got deferred.
- **Layer 3 — Hard pause for high-impact additions:** auth, database swaps, payment, new personal-data types, new communication channels → mandatory mini-`/vd-plan` with 3 options. No silent auto-add.

**Why this matters:** Feature creep is the silent killer of layman projects. The IA is generous and suggests "you could also add..." — without containment, scope doubles in a week.

#### 3. Recognition & Pause (embedded rituals)

Added in `SKILL.md`.

Building alone is exhausting. The framework now has three small embedded rituals:

1. **Factual micro-recognition every 3 sub-tasks.** One sentence, no emoji, no enthusiasm: "3 sub-tasks of Phase 6 done: login, signup, dashboard. 5 left."
2. **Pause prompt at phase gates.** When a phase closes and the next one is about to start, the IA acknowledges the work honestly and asks if the user wants to take a real break — 15 minutes, 1 day, 1 week — before starting the next phase.
3. **Anti-guilt for "how much longer?" questions.** Never "you should have started earlier" or "definitely 2 more weeks." Always interval ranges from the time table, framed as realistic, never as promise.

**Burnout signal detection:** if the user's responses become very short ("ah sei lá", "seja lá o que for", "ah tanto faz") or they keep repeating the same errors, the framework offers `/vd-kill` as a legitimate option — without lecturing.

### Backward compatibility

- All new behaviors are passive — they surface only when relevant. No existing command flow changed.
- Time estimates only appear in layman mode; technical mode users see no change in `/vd-start` flow.
- Anti-creep guard adds new behavior inside existing commands, doesn't change command signatures.
- Recognition rituals are factual, one-line, opt-out by user request ("just code, no chat").

## 🇧🇷 Português

**Data do release:** 2026-07-09
**Escopo:** VibeDev core (v1.3.0)

No v3.1.0 a gente resolveu três problemas de "não entendo" pro leigo (jargão, custo, kill). Esse release v3.2.0 resolve três problemas de "tô exausto / perdi o controle / não sei quanto falta".

Essas features são passivas — só aparecem quando relevantes. Não mudam nenhum comando existente, só adicionam comportamento dentro deles.

### O que tem nesse release

#### 1. Estimativas de Tempo Calibradas por Categoria

Arquivo novo: `vibedev/references/tempo-construcao.md`

Carregado durante `/vd-start` em modo leigo **antes** do diagnóstico formal. As 9 categorias mais comuns ganham intervalo realista de tempo:

| Categoria | Leigo (4h+/dia) | Leigo (horas vagas) | Dev |
|---|---|---|---|
| Landing page (1 página estática) | 1-3 dias | 1-2 semanas | 2-8h |
| Site institucional (3-10 páginas) | 2-4 semanas | 1-2 meses | 1-3 semanas |
| App interno / dashboard privado | 1-2 meses | 2-6 meses | 2-6 semanas |
| Bot em mensageria (Telegram/WhatsApp + LLM) | 3-8 semanas | 2-4 meses | 1-2 semanas |
| SaaS B2C público (freemium, login, pagamento) | 3-12 meses | 6-24 meses | 2-6 meses |
| App mobile (iOS + Android na loja) | raramente termina | raramente termina | 3-9 meses |
| Marketplace (estilo Uber/iFood) | não tenta solo | não tenta solo | 6-18 meses |
| App com IA generativa (chatbot, gerador) | 1-4 meses (MVP) | 2-8 meses | 2-6 semanas |

**Enquadramento honesto:** "pessoas com ideias parecidas com a sua geralmente levam X a Y semanas trabalhando Z horas por dia. Não é regra firme — é pra você não achar que tá atrasado quando não tá."

**Mitigação de vazamento:** se o leigo chega com expectativa de timeline de reel de Instagram e claramente não tem o tempo, o framework oferece `/vd-kill` como opção legítima **antes** de começar, não 90 dias depois de um projeto travado.

#### 2. Guarda Anti-Feature-Creep

Arquivo novo: `vibedev/references/anti-creep.md`

Contenção em 3 camadas que mantém o escopo honesto:

- **Camada 1 — Scope check pré-build:** antes de escrever código de uma sub-tarefa, a IA compara intenção com `anti-escopo`, critério de pronto da sub-tarefa ativa, e lista de sub-tarefas restantes. Não bate → backlog, não código.
- **Camada 2 — Detecção em tempo real:** ideias novas durante `/vd-build` (seja da IA ou do usuário) são reconhecidas como boas, registradas no backlog automaticamente, e execução continua na sub-tarefa planejada. Fim da sub-tarefa menciona o que foi deferido.
- **Camada 3 — Pause explícita em adições de alto impacto:** auth, troca de banco, pagamento, novo tipo de dado pessoal, novo canal de comunicação → mini-`/vd-plan` obrigatório com 3 opções. Sem auto-add silencioso.

**Por que isso importa:** Feature creep é o matador silencioso de projetos de leigo. A IA é generosa e sugere "você podia também adicionar..." — sem contenção, o escopo dobra em uma semana.

#### 3. Reconhecimento & Pausa (rituais embutidos)

Adicionado na `SKILL.md`.

Construir sozinho é exaustivo. O framework agora tem 3 pequenos rituais embutidos:

1. **Micro-reconhecimento factual a cada 3 sub-tarefas.** Uma frase, sem emoji, sem entusiasmo artificial: "3 sub-tarefas da Fase 6 prontas: login, cadastro, dashboard. Faltam 5."
2. **Pausa explícita em gates de fase.** Quando uma fase fecha e a próxima está prestes a começar, a IA reconhece o trabalho honestamente e pergunta se a pessoa quer dar uma pausa real — 15 min, 1 dia, 1 semana — antes de começar a próxima fase.
3. **Anti-culpa em perguntas "quanto falta?"**. Nunca "você devia ter começado antes" ou "mais 2 semanas com certeza". Sempre intervalos da tabela de tempo, enquadrados como realista, nunca como promessa.

**Detecção de sinal de burnout:** se as respostas do usuário ficam muito curtas ("ah sei lá", "tanto faz", "ah seja lá o que for") ou ele repete os mesmos erros, o framework oferece `/vd-kill` como opção legítima — sem sermão.

### Compatibilidade

- Todas as novas features são passivas — só aparecem quando relevantes. Nenhum comando existente mudou.
- Estimativa de tempo só aparece em modo leigo; usuários técnicos não veem mudança no fluxo de `/vd-start`.
- Guarda anti-creep adiciona comportamento novo dentro dos comandos existentes, não muda assinaturas.
- Reconhecimento é factual e curto, opt-out por pedido do usuário ("só código, sem conversa").
